Project Overview

Designed and deployed a comprehensive centralized monitoring and management server for all production PostgreSQL databases across the organization. This solution consolidated monitoring, logging, health checks, and alerting into a single unified platform, significantly improving operational efficiency and reducing manual workload.

The monitoring server provides real-time visibility into database health, performance metrics, and potential issues across all production instances. It automatically collects logs, performs health checks, generates alerts, and provides intuitive dashboards for quick assessment of the entire database infrastructure.

Problem Statement

Before implementing the centralized monitoring server, the team faced several operational challenges:

  • Manual checking of 50+ production databases consuming significant time daily
  • Fragmented monitoring tools across different database instances
  • Delayed detection of performance issues and anomalies
  • Logs scattered across multiple servers making troubleshooting difficult
  • No unified view of overall database infrastructure health
  • Reactive approach to issues rather than proactive monitoring
  • Inconsistent health check procedures across databases
  • Alert fatigue from poorly configured individual database alerts
Business Impact

The lack of centralized monitoring resulted in slower incident response times, increased operational overhead, and potential service disruptions that could have been prevented with proactive monitoring. DBAs spent 4-5 hours daily on manual checks that could be automated.

Solution Architecture

1. Centralized Logging System

Implemented comprehensive log aggregation and analysis:

  • Configured all production databases to forward logs to central server
  • Set up log rotation and retention policies (30 days online, 6 months archive)
  • Implemented log parsing and categorization (errors, warnings, slow queries)
  • Created searchable log repository for quick troubleshooting
  • Automated log analysis to identify recurring issues and patterns
  • Integrated with alerting system for critical log events
# Log forwarding configuration
logging_collector = on
log_directory = 'pg_log'
log_filename = 'HOSTNAME-%Y-%m-%d_%H%M%S.log'
log_statement = 'DDL'
log_min_duration_statement = 10000 # Log queries > 1s
# Central log collection script
rsync -avz --remove-source-files \
user@db-server:/var/log/postgresql/ \
/monitoring-server/logs/db-server/

2. Automated Health Checks

Developed comprehensive health check framework:

  • Connection pool status and availability
  • Replication lag monitoring (< 5 seconds threshold)
  • Disk space utilization (alert at 80%, critical at 90%)
  • Database bloat detection and reporting
  • Lock monitoring and deadlock detection
  • Long-running query identification (> 30 minutes)
  • CPU and memory usage tracking
  • Backup status verification (last successful backup time)
  • Vacuum and analyze status
  • Index usage statistics

# Automated health check script (runs every 5 minutes)
#!/bin/bash

for db in $(cat production_databases.txt); do
# Check connection
psql -h $db -U monitor -c "SELECT 1" > /dev/null 2>&1
if [ $? -ne 0 ]; then
send_alert "Connection failed: $db"
fi

# Check replication lag
lag=$(psql -h $db -U monitor -tAc \
"SELECT EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp()))")
if [ "$lag" -gt 5 ]; then
send_alert "Replication lag: $db ($lag seconds)"
fi

# Check disk space
disk=$(ssh $db "df -h /var/lib/postgresql | tail -1 | awk '{print \$5}' | sed 's/%//'")
if [ "$disk" -gt 90 ]; then
send_critical_alert "Disk space critical: $db ($disk%)"
fi
done
