Project Overview

Led a large-scale migration project moving critical PostgreSQL databases ranging from 500 GB to 4 TB from on-premise Data Center infrastructure to Amazon Web Services (AWS) cloud platform. This strategic initiative was designed to leverage cloud scalability, improved disaster recovery capabilities, and enhanced operational flexibility while ensuring zero data loss and minimal service disruption.

The project involved comprehensive migration planning, load testing, storage and I/O performance optimization on AWS, and carefully orchestrated phased cutover to ensure business continuity. Successfully migrated multiple mission-critical databases supporting key business applications with improved performance metrics post-migration.

Problem Statement

The existing on-premise infrastructure presented several limitations:

  • Limited scalability requiring lengthy hardware procurement cycles
  • High capital expenditure for infrastructure upgrades
  • Disaster recovery capabilities constrained by physical infrastructure
  • Manual scaling processes unable to handle sudden traffic spikes
  • Aging hardware approaching end-of-life requiring replacement
  • Limited geographic redundancy options
  • High operational overhead for infrastructure maintenance
  • Difficulty in implementing modern DevOps practices
Business Drivers

The migration to AWS was driven by needs for better scalability, reduced capital expenses, improved disaster recovery, and the ability to leverage cloud-native services for enhanced application capabilities. The migration needed to be completed within 6 weeks with zero data loss.

Optimized AWS storage for database workloads:

Systematic approach to minimize risk:

  • Each wave validated before proceeding to next
  • Lessons learned incorporated into subsequent waves

Migration Workflow

For Each Database:

  • Day -7: Initial full backup using pg_basebackup
  • Day -5: Transfer backup to AWS over VPN
  • Day -3: Restore database on AWS EC2 instance
  • Day -2: Set up streaming replication from Data Center to AWS
  • Day -1: Monitor replication lag and validate data consistency
  • Day 0 (Cutover):
    • Place application in maintenance mode
    • Wait for replication lag to reach zero
    • Promote AWS database to primary
    • Update application connection strings
    • Perform smoke tests
    • Resume application operations
  • Day +1 to +7: Monitor performance and optimize as needed

# Migration workflow script
#!/bin/bash

DB_NAME=$1
DC_HOST="datacenter-db.local"
AWS_HOST="ec2-xx-xxx-xxx-xxx.compute.amazonaws.com"

# Step 1: Create base backup
pg_basebackup -h $DC_HOST -D /backup/$DB_NAME -Fp -Xs -P

# Step 2: Transfer to AWS
rsync -avz --progress /backup/$DB_NAME/ $AWS_HOST:/var/lib/postgresql/14/main/

# Step 3: Configure replication
ssh $AWS_HOST "cat > /var/lib/postgresql/14/main/recovery.conf << EOF
primary_conninfo = 'host=$DC_HOST port=5432 user=replicator'
restore_command = 'cp /backup/wal/%f %p'
standby_mode = on
EOF"

# Step 4: Start AWS instance in standby mode
ssh $AWS_HOST "pg_ctl start -D /var/lib/postgresql/14/main"

echo "Migration setup complete. Monitor replication lag before cutover."

  • Implemented EBS volume striping for largest databases

Instance Optimization

  • Leveraged enhanced networking for improved performance
  • Configured instance placement groups for low-latency communication
  • Implemented auto-scaling groups for read replicas

PostgreSQL Tuning

  • Optimized memory settings for r5 instance types
  • Tuned checkpoint settings for AWS storage characteristics
  • Configured connection pooling with pgBouncer
  • Optimized vacuum and analyze schedules
  • Implemented query result caching strategies

Key Challenges & Solutions

Challenge 1: Large Database Transfer Times

Problem: Transferring multi-terabyte databases over network taking 24+ hours.

Solution: Used AWS Snowball for initial data transfer of largest databases, then set up replication for continuous sync. Reduced transfer time by 70%.

Challenge 2: Network Bandwidth Limitations

Problem: Limited bandwidth between Data Center and AWS causing bottlenecks.

Solution: Established AWS Direct Connect for dedicated high-bandwidth connection, implemented compression for data transfer, and scheduled large transfers during off-hours.

Challenge 3: I/O Performance Differences

Problem: Some queries performed differently on AWS due to storage characteristics.

Solution: Conducted thorough performance testing, adjusted PostgreSQL I/O parameters, provisioned additional IOPS where needed, and optimized query plans for AWS environment.

Challenge 4: Application Latency Concerns

Problem: Some on-premise applications experienced increased latency connecting to AWS databases.

Solution: Implemented connection pooling to reduce connection overhead, deployed read replicas closer to applications, and optimized network routing. For critical applications, planned gradual application migration to AWS.

Lessons Learned

  • Comprehensive load testing in AWS environment is critical before production migration
  • Storage performance characteristics on AWS differ from traditional SAN - tune accordingly
  • Replication-based migration minimizes downtime for large databases
  • Phased approach allows learning and adaptation between waves
  • Close coordination with application teams prevents surprises
  • Post-migration monitoring and optimization are essential
  • Documentation of cloud-specific configurations aids future troubleshooting
  • Consider application migration alongside database migration for optimal performance
